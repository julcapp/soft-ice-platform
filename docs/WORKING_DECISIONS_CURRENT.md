# Soft ICE — текущие согласованные решения

Статус: Approved  
Дата фиксации: 2026-06-26  
Проект: Soft ICE Platform / бренд «У Тимоши»

## 1. Общая архитектурная позиция

Soft ICE рассматривается не как отдельный Telegram-бот, CRM или интерфейс вендингового аппарата, а как единая цифровая платформа управления продажами, операциями, коммуникациями, лояльностью и аналитикой.

Бренд «У Тимоши» является первым продуктовым брендом на базе платформы Soft ICE.

CRM больше не считается центром архитектуры. CRM — это одно из приложений поверх платформенного ядра.

## 2. Принципы продукта

Приняты следующие принципы:

1. Documentation First — сначала фиксируется решение, затем выполняется разработка.
2. Platform First — новая функция должна быть полезна платформе, а не только одному экрану.
3. Zero Friction UX — пользователь должен делать минимум действий.
4. Progressive Engagement — сначала покупка и ценность, затем вовлечение в клуб.
5. Progressive Profiling — данные собираются постепенно и только тогда, когда они нужны.
6. Omnichannel First — Telegram, MAX, VK, сайт, терминал и Seller App являются равноправными каналами.
7. Event Driven UI — интерфейс отображает фактическое состояние системы, а не предполагаемый таймер.
8. Evidence over Assumption — подтверждённые действия важнее формальных отметок.
9. Single Source of Truth — данные не дублируются без необходимости.
10. Single Source of Visual Truth — визуальные активы создаются один раз и используются во всех каналах.

## 3. Язык бренда и UX

Принято решение убрать слово «регистрация» из пользовательских экранов.

Запрещённые формулировки для пользовательского интерфейса:

- Зарегистрируйтесь
- Создайте аккаунт
- Заполните профиль
- Введите персональные данные

Рекомендуемые формулировки:

- Получить скидку
- Вступить в Клуб Тимоши
- Получить бонусы
- Получить подарок
- Получать специальные предложения

Главное правило интерфейса:

> Каждый экран должен помогать человеку сделать следующий естественный шаг, а не собирать данные ради системы.

## 4. Customer Journey терминала v1.0

Утверждён базовый сценарий покупки через вендинговый аппарат:

1. Приветствие: «Собери своё мороженое».
2. Вкус дня.
3. Выбор сиропа.
4. Выбор топпинга.
5. Проверка заказа.
6. Оплата.
7. Приготовление.
8. Получение мороженого.
9. Фискальный чек.
10. Спасибо за покупку.
11. Предложение Клуба Тимоши.
12. Постпокупочный экран.

Ключевое продуктовое решение:

> Покупатель не должен быть обязан вводить телефон для совершения покупки. Главная цель автомата — продать продукт. Телефон нужен для получения скидки, бонусов, чека на телефон и участия в клубной механике.

## 5. Вкус дня

В аппарате на текущем этапе используется один вид смеси.

Поэтому терминал не предлагает пользователю свободный выбор вкуса. Он показывает активный «Вкус дня», заданный в системе управления аппаратом.

Логика:

1. Администратор в панели управления выбирает загруженную смесь.
2. Система управления передаёт активный вкус на терминал.
3. На терминале активен только текущий вкус.
4. Остальные вкусы могут отображаться как недоступные или скрываться в зависимости от UI-решения.

В будущем, если аппарат будет поддерживать 2–3 смеси, экран сможет показывать несколько активных вкусов без переработки архитектуры.

## 6. Сиропы

Принято: в аппарате предусмотрено 3 вида сиропа.

Экран выбора сиропа:

- Заголовок: «Добавь любимый сироп».
- Карточки: три сиропа + «Без сиропа».
- Один выбор.
- Названия и изображения загружаются из панели управления.
- Если сироп закончился, карточка становится недоступной.

Правило:

> Терминал никогда не предлагает пользователю то, чего физически нет в аппарате.

## 7. Топпинги

Принято: в аппарате предусмотрено 3 вида топпинга.

Экран выбора топпинга:

- Показываются варианты топпинга на мороженом в стаканчике.
- Не использовать рожок, потому что продукт продаётся в стаканчике.
- Добавляется вариант «Без топпинга».
- Названия, изображения и доступность приходят из панели управления.

Изображения топпингов должны быть высокого качества и использоваться повторно в терминале, сайте, Telegram Mini App, MAX Mini App, VK Mini App, CRM и Seller App.

## 8. Оплата

Утверждена модульная архитектура оплаты:

PaymentScreen → Payment API → Payment Adapter.

Поддерживаемые сценарии:

- Оплата картой.
- Оплата по QR / СБП.
- Получить оплаченный заказ.

Экран оплаты не содержит бизнес-логики. Он только отображает доступные способы оплаты и передаёт выбор в Payment API.

## 9. Фискальный чек

После успешной оплаты показывается экран фискального чека.

Требования:

- Заголовок: «Ваш фискальный чек».
- QR-код с фискальными данными от ОФД.
- Текст: «Отсканируйте QR-код или получите чек на телефон».
- Текст: «Экран закроется через 15 секунд».
- На экране работает обратный отсчёт.
- Кнопки: «Получить на телефон» и «Пропустить».

Если пользователь выбирает получение чека на телефон, открывается экран ввода номера с цифровой клавиатурой.

## 10. Приготовление и выдача

Интерфейс приготовления должен быть событийным.

Терминал отображает реальные состояния автомата:

- PreparationStarted
- IceCreamDispensed
- SyrupDispensed
- ToppingDispensed
- ProductReady
- DoorOpened
- DoorClosed
- ProductTaken

Интерфейс не должен показывать «готово», если оборудование ещё не завершило операцию.

## 11. Actor Model

Принято архитектурное решение Actor Model.

Любой участник системы, выполняющий действие, является Actor:

- Customer
- Vending Machine
- Seller
- Mobile Point
- Partner
- Franchisee
- Service Engineer
- Operator
- Administrator
- System

Каждое действие Actor превращается в событие и попадает в Audit Log, Analytics, CRM и при необходимости в Communication Hub.

## 12. Entry Points

Пользователь не должен искать бота вручную.

Входы в экосистему осуществляются через контекстные точки входа:

- QR-код на аппарате.
- NFC-метка на аппарате.
- Deep Link.
- Сайт.
- VK-сообщество.
- Telegram.
- MAX.
- Продавец.
- Реферальная ссылка.

QR и NFC должны содержать Deep Link и передавать контекст: источник, точку продаж, кампанию, автомат или продавца.

QR на стаканчике признан интересной идеей, но отложен до отдельной UX-проработки, потому что после получения мороженого у пользователя могут быть заняты руки.

## 13. Communication & Community

VK.com включается в архитектуру как часть Community Platform.

Разделение каналов:

- Transaction Channels: терминал, сайт, Telegram Mini App, MAX Mini App, Seller App.
- Communication Channels: VK, Telegram Channel, MAX Channel, Email, Push, SMS.

Для коммуникаций закладывается Communication Hub:

- Notification Service
- Template Engine
- Telegram Service
- MAX Service
- VK Service
- Email Service
- SMS Service
- Campaign Manager

Собственный SMTP-сервер на старте не поднимается. Для email следует использовать внешнего провайдера.

## 14. Phone Verification и SMSC.RU

Телефон используется как один из identity-каналов, но не должен блокировать быструю покупку.

Для сценариев клуба, скидки, бонусов и получения чека телефон подтверждается через SMSC.RU.

Основной способ подтверждения — обратный звонок для снижения стоимости.

Планируемые события:

- PhoneVerificationStarted
- PhoneVerificationCodeSent
- PhoneVerificationSucceeded
- PhoneVerificationFailed
- PhoneLinkedToCustomer

## 15. Media Library и Design System

Принято решение создать единую медиатеку Soft ICE.

Один визуальный актив создаётся один раз и повторно используется во всех каналах.

Новые документы для развития:

- MEDIA_STANDARD.md
- BRAND_PHOTO_BIBLE.md
- DESIGN_SYSTEM.md
- ASSET_LIBRARY.md
- UI_COMPONENTS.md
- DESIGN_TOKEN_STANDARD.md

Каждый следующий экран должен проектироваться так, чтобы его можно было без изменений превратить в:

- React-компонент;
- XML-макет терминала;
- экран Mini App.

## 16. ЮKassa

ЮKassa запросила новые отдельные полноэкранные скриншоты после изменения названия бота.

Требуется подготовить реальные скриншоты из Telegram, а не коллаж:

- до перехода в Mini App;
- после перехода в Mini App;
- с видимым названием бота;
- с карточками товаров/услуг;
- с ценами в рублях;
- с описаниями;
- несколькими примерами товаров/услуг.

Для проверки предлагаются позиции:

1. Мороженое «Вкус дня» — 130 ₽.
2. Пополнение клубного счёта / Клуб Тимоши — 300 ₽.
3. Подарочный сертификат — 500 ₽.

2. Подготовить комплект для ЮKassa.

4. Сделать новые отдельные полноэкранные скриншоты.
5. Отправить комплект в ЮKassa.
